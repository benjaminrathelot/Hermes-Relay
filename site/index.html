Communication when networks fail

Short encrypted messages that can survive a broken world.

Hermes Relay is an offline-first messaging protocol designed for war zones, weather disasters, infrastructure collapse, and fractured regions where some places still have connectivity and others have none.

One sender. One recipient. No central server required. A message can cross several relays through LAN, Wi-Fi, partial Internet bridges, file import/export, or the physical movement of a laptop, phone, relay box, or storage device carrying encrypted envelopes until the recipient eventually receives them.

When normal apps fail

Hermes Relay exists for broken conditions, not ideal ones.

If sender and recipient never overlap online, if towers are down, if only one district still has Internet, or if the message must travel by human movement, the protocol still has a job to do.

No live path required

Delay is acceptable. Silence is not.

Hermes Relay assumes that a message may need to wait on a relay, a workstation, or a carried device until the next contact opportunity appears.

How It Works

One encrypted envelope. Several possible hops.

The protocol does not need one continuous network path from origin to destination. It only needs enough intermittent contact for relays and devices to exchange opaque envelopes over time.

01

Compose

A sender creates a short encrypted message for exactly one recipient, signs it, and solves the attached proof of work.

02

Store

The message is stored locally in a client or relay. Plaintext is never sent across the transport fabric.

03

Carry

Whenever a new contact becomes possible, the envelope can move over LAN, Wi-Fi, Internet relay, or file handoff.

04

Relay Again

The next node stores it, deduplicates it, and later forwards it onward. The process repeats until it reaches the recipient side.

05

Deliver

The intended recipient decrypts the envelope locally. The sender and recipient never need to be online together.

Relay Fabric

A communication fabric made from whatever links still exist.

Some locations may have Internet. Others may only have local Wi-Fi. Others may require physical transport of a device or bundle. Hermes Relay is built for all of these links to coexist inside one propagation model.

Online zones Local sync or Internet bridging possible
Dark zones Traffic continues later by relay or physical carry
Live contact

LAN, Wi-Fi, hotspot, bridge relay

Two relays or clients meet, exchange inventories, request what is missing, verify envelopes, and keep going.

Deferred contact

File export, USB, carried phone or laptop

An encrypted bundle leaves one island, travels physically, and is imported later into another island’s relay or desktop app.

Mixed geography

Some places online, others offline

Internet-connected relays can bridge long distances while isolated areas continue to move traffic by local or physical means.

Delivery Over Time

The protocol accepts delay in exchange for survivability.

Hermes Relay is useful precisely because communication can continue even when delivery is not immediate.

Hour 0

Message created

A field operator creates one short encrypted message for one recipient.

Hour 3

Local relay sync

A nearby relay receives the envelope over LAN and keeps it in store.

Day 1

Bundle ferried physically

A laptop or phone carrying bundles crosses into another district and imports them there.

Day 2

Bridge zone online

Day 3

Recipient side receives

The recipient’s device eventually decrypts the message locally.

Project Scope

Protocol first. Real tools second. Mobile field use next.

The repository already includes a serious V1 protocol baseline: canonical envelopes, crypto, relay runtime, CLI, simulator, Python wrapper, and a local desktop control surface.

The next major step is mobile integration, so that smaller devices can store, carry, and relay high volumes of short encrypted messages in the same multi-hop propagation fabric.

Current

Portable C core and relay baseline

Protocol, storage, relay service, simulator, CLI, Python wrapper, and desktop shell.

Next

Mobile participation in the same protocol

Phones and small devices acting as readers, senders, carriers, and opportunistic relays.

Later

More transports, same envelope

Bluetooth, Wi-Fi Direct, QR, LoRa, OpenWRT routers, and broader independent implementation interoperability.
