Field communication continuity

Offline messaging for crisis communication when normal networks fail.

Hermes Relay is an offline-first messaging protocol for degraded environments: war zones, weather disasters, infrastructure collapse, and split regions where some places still have connectivity and others have none.

One sender. One recipient. One encrypted envelope. No central server required. A message can cross several relays through LAN, Wi-Fi, partial Internet bridges, file import/export, or the physical movement of a laptop, phone, relay box, or storage device carrying encrypted envelopes until the recipient eventually receives them.

When normal apps fail

Hermes Relay exists for broken conditions, not ideal ones.

If sender and recipient never overlap online, if towers are down, if only one district still has Internet, or if the message must travel by human movement, the protocol still has a job to do.

No live path required

Delay is acceptable. Silence is not.

Hermes Relay assumes that a message may need to wait on a relay, a workstation, or a carried device until the next contact opportunity appears.

What The User Actually Does

For the end user, this should feel simpler than the network behind it.

The user should not have to route traffic manually. They create an identity, add contacts, write a short message, and read messages that arrive for them. The relay layer is responsible for the spider work in the background.

Identity

Each participant has a public/private key pair.

Every message is created by one sender identity and encrypted for one recipient identity. Hermes Relay is not group chat, not broadcast social posting, and not a public feed.

Message model

One message. One sender. One recipient.

The protocol keeps the unit of communication narrow and auditable. A user writes a short text and addresses it to a single contact.

Relay behavior

The relay is the opportunistic operator.

Relays discover contact opportunities, exchange inventories, request missing envelopes, deduplicate traffic, store what is valid, and keep diffusing messages without asking the user to micromanage each hop.

Practical effect

The user carries messages by using the system, not by understanding routing.

As future mobile apps arrive, even small devices will be able to carry and relay large numbers of short encrypted messages while the end user focuses on communication, not network topology.

How It Works

One encrypted envelope. Several possible hops. Intelligent relay behavior in between.

The protocol does not need one continuous network path from origin to destination. It only needs enough intermittent contact for relays and devices to exchange opaque envelopes over time.

01

Compose

A sender creates a short encrypted message for exactly one recipient, signs it, and solves the attached proof of work.

02

Store

The message is stored locally in a client or relay. Plaintext is never sent across the transport fabric.

03

Diffuse

The relay does the opportunistic work: inventory exchange, missing-envelope requests, validation, storage, prioritization, and onward propagation.

04

Carry

Whenever new contact becomes possible, the envelope can move over LAN, Wi-Fi, Internet relay, file handoff, or a physically carried device.

05

Relay Again

The next node stores it, deduplicates it, and later forwards it onward. The process repeats until it reaches the recipient side.

06

Deliver

The intended recipient decrypts the envelope locally. The sender and recipient never need to be online together.

Relay Fabric

A communication fabric made from whatever links still exist.

Some locations may have Internet. Others may only have local Wi-Fi. Others may require physical transport of a device or bundle. Hermes Relay is built for all of these links to coexist inside one propagation model.

Frequently Asked Questions

What people usually need to understand first.

Hermes Relay is easiest to understand once the user model and relay model are separated clearly.

Does Hermes Relay require Internet access?

No. It can work entirely without Internet. Partial Internet only helps bridge some regions more quickly when available.

What does the end user actually do?

Create an identity, add contacts, write a short message to one recipient, and read messages addressed to that identity.

What do relays do automatically?

Relays exchange inventories, request missing envelopes, verify messages, deduplicate traffic, store valid envelopes, and keep diffusing them opportunistically.

Can delivery take hours or days?

Yes. That is normal in the design. The protocol is meant to preserve communication across fragmented geography and intermittent contact, not guarantee instant delivery.

Is this group chat?

No. Hermes Relay V1 is intentionally narrow: one sender, one recipient, one short encrypted message, no attachments, no public feed.

Why is mobile integration important?

Because the long-term value is strongest when phones and other small devices can participate directly as senders, receivers, carriers, and relays in the same protocol.

Project Scope

Protocol first. Smart relay behavior next. Mobile field use after that.

The repository already includes a serious V1 protocol baseline: canonical envelopes, crypto, relay runtime, CLI, simulator, Python wrapper, and a local desktop control surface.

The next major step is mobile integration, so that smaller devices can store, carry, and relay high volumes of short encrypted messages in the same multi-hop propagation fabric.

Current

Portable C core and relay baseline

Protocol, storage, relay service, simulator, CLI, Python wrapper, and desktop shell.

Next

Mobile participation in the same protocol

Phones and small devices acting as readers, senders, carriers, and opportunistic relays.

Later

More transports, same envelope

Bluetooth, Wi-Fi Direct, QR, LoRa, OpenWRT routers, and broader independent implementation interoperability.
